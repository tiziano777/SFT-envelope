       - **s3://**: S3StorageWriter (stub: NotImplementedError with guidance)
       - **nfs://**: NFSStorageWriter (stub: NotImplementedError with guidance)
       - **worker://**: Artifact on Worker node (not copied to Master)
       - **master://**: Artifact on Master node (for merged checkpoints)

    3. **Configuration (via .env):**
       ```bash
       # Map URI prefix to storage backend
       URI_PREFIX_FILE=local       # file:// → LocalStorageWriter
       URI_PREFIX_S3=s3             # s3:// → S3StorageWriter (TBD)
       URI_PREFIX_NFS=nfs           # nfs:// → NFSStorageWriter (TBD)
       ```

    4. **During Handshake:**
       - Worker sends checkpoint URIs as file:// (local GPU storage)
       - Master stores URI in Checkpoint node (not copying data)
       - Assumption: checkpoints remain on Worker for RESUME strategy

       - Merge script reads source checkpoints (via their URIs)
       - Writes merged checkpoint to local disk
       - Generates new uri: master:///merged/ckp_id/model.bin
       - Reports back to Master with merged checkpoint URI

    6. **Extensibility:**
       - Custom backends can implement BaseStorageWriter ABC
       - RegisterWriter pattern allows new backends without modifying URIResolver

    3. **Architectural Layers:**
       ```
       Layer 1: Shared Contracts (Phase 1)
       ├─ ConfigSnapshot, ConfigHasher, DiffEngine

       └─ Transport envelopes: HandshakeRequest/Response, CheckpointPush, SyncEvent

       ├─ Neo4j 5.x with 5 UNIQUE constraints
       ├─ APOC triggers for timestamps and orphan validation
       └─ ExperimentRepository with idempotent Cypher queries

       Layer 3: Observability (Phase 3)
       ├─ OpenTelemetry SDK
       ├─ FastAPI auto-instrumentation
       └─ Phoenix UI for trace visualization

       Layer 4: Master API (Phase 4)
       ├─ 5 REST endpoints
       ├─ LineageController with strategy logic
       └─ Error handlers with semantic status codes

       Layer 5: Storage (Phase 5)
       ├─ URIResolver for artifact dispatch
       └─ Pluggable backends (file, s3, nfs, worker, master)

       Layer 6: Worker (Phase 6)
       ├─ WorkerDaemon for handshake and async push
       ├─ Connection layer (HTTP, SSH stubs)
       └─ AsyncPusher with exponential backoff

       Layer 7: Generator (Phase 7)
       ├─ inject_worker_middleware() copies worker code to scaffold
       ├─ run.sh.j2 orchestrates daemon lifecycle
       └─ MergeTechnique plugin for checkpoint merging

       Layer 8: Datamix (Phase 8)
       └─ Multi-source dataset support (independent of lineage)

       Layer 9: Testing (Phase 9)
       ├─ E2E test suite with real Neo4j
       └─ All 4 strategies tested (NEW, RESUME, BRANCH, RETRY)
       ```

    4. **Data Flow Diagram:**
       Show how a checkpoint flows from Worker to Master:
       ```
       [Worker] train.py epoch 5, run 2
           ↓ (writes checkpoint to disk)
       [Worker] daemon.pusher observes lineage/to_transfer/
           ↓ (reads checkpoint metadata)
       [Worker] queues CheckpointPush event
           ↓ (async, with retry)
       [Network] HTTP POST /checkpoint_push
           ↓
       [Master] FastAPI validates request
           ↓
       [Master] LineageController processes push
           ↓ (checks consistency)
       [Master] Neo4j creates Checkpoint node
           ↓
       [Master] returns 200 + checkpoint_id
           ↓
       [Network] HTTP response
           ↓
       [Worker] pusher deduplicates (event_id) and logs to transfer_log.jsonl
       ```

    5. **Strategy Detection Logic (Decision Tree):**
       Explain how handshake determines strategy:
       ```
       Handshake: find_experiment_by_hashes(config_hash, code_hash, req_hash)
           ├─ Found with all hashes matching? → RESUME (same config, same code)
           ├─ Found but config_hash differs? → BRANCH (config changed, new direction)
           ├─ Found but code_hash differs? → BRANCH (train.py changed)
           ├─ Not found + seed/run specified? → RETRY (repeat with different seed)
           └─ Not found + fresh start? → NEW (fresh experiment)
       ```

    6. **Async Push with Retry:**
       Explain exponential backoff:
       - Attempt 1: immediate
       - Attempt 2: 2s delay
       - Attempt 3: 4s delay
       - ...
       - Attempt N: min(2^N, 300s) = max 5 min
       - Give up after 10 attempts or 30 min timeout

    7. **Failure Modes & Degradation:**
       - **Master unavailable during handshake**: Warning but continue (degraded mode, no exp_id)
       - **Master unavailable during training**: Daemon queues pushes locally; syncs when Master back
       - **Network timeout**: Exponential backoff retry; training unaffected
       - **Circular lineage detected**: Master rejects with 409 Conflict

    8. **Security Model:**
       - **Auth**: X-API-Key on all endpoints (reject 401 if missing)
       - **Validation**: All inputs validated (Pydantic models at API boundary)
       - **Trust**: Master trusts Worker during handshake (must be on same network or VPN)
       - **Audit**: transfer_log.jsonl immutable append-only audit trail

    9. **Performance Characteristics:**
       - Handshake: < 100ms (Neo4j query + hash comparison)
       - Checkpoint push: < 500ms (Neo4j transaction + storage check)
       - Worker daemon overhead: < 5% CPU, < 50MB RAM (idle)

    10. **Next Steps / Further Reading:**
        - Detailed API: docs/lineage/api-reference.md
        - Schema: docs/lineage/schema.md
        - Troubleshooting: docs/lineage/troubleshooting.md
