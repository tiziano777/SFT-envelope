## Category: Build & Environment

### R-BUILD-1: Python Packaging
- **Title**: Generate requirements.txt from codebase imports
- **Acceptance**:
  - `requirements.txt` lists all top-level deps (torch, pydantic, click, rich, jinja2, pyyaml, etc)
  - Pins versions (e.g., `pydantic>=2.0,<3.0`)
  - Can `pip install -r requirements.txt` without errors
  - `python -c "import envelope; print(envelope.__version__)"` succeeds
- **Priority**: CRITICAL
- **Phase**: 1

### R-BUILD-2: Makefile Command Suite
- **Title**: Create standard dev Makefile
- **Acceptance**:
  - `make install` → pip install -e .
  - `make test` → pytest envelope/
  - `make lint` → ruff check envelope/
  - `make clean` → rm -rf __pycache__/, .pytest_cache, etc
  - All commands idempotent
- **Priority**: HIGH
- **Phase**: 1

### R-BUILD-3: .gitignore Standardization
- **Title**: Exclude runtime/cache artifacts from git
- **Acceptance**:
  - .cache/ ignored
  - output/ ignored
  - .venv/ ignored
  - *.pyc, __pycache__ ignored
  - .DS_Store ignored
  - .env* files ignored
- **Priority**: MEDIUM
- **Phase**: 1

## Category: Scaffold Generation

### R-SCAFFOLD-1: train.py Template
- **Title**: Executable training script template in setup folder
- **Acceptance**:
  - Template in `envelope/generators/templates/train.py.jinja2`
  - Rendered to `setup_{name}/train.py`
  - Reads `config.yml` from same folder
  - Loads model, dataset, initializes trainer per framework
  - No hardcoded hyperparams — all from config
  - Can be executed: `python train.py` or `bash run.sh`
- **Priority**: CRITICAL
- **Phase**: 2

### R-SCAFFOLD-2: prepare.py Template
- **Title**: Data preprocessing script template
- **Acceptance**:
  - Template in `envelope/generators/templates/prepare.py.jinja2`
  - Reads `dataset` section from config.yml
  - Loads recipe (if exists), applies preprocessing
  - Outputs preprocessed data to `.cache/` directory
  - Idempotent — safe to re-run
  - Supports HuggingFace datasets + local files
- **Priority**: CRITICAL
- **Phase**: 2

### R-SCAFFOLD-3: config.yml Atomicity
- **Title**: Config file contains 100% of training logic
- **Acceptance**:
  - No runtime env vars override hyperparams (except debug flags)
  - All technique args in `training.technique_args`
  - Hardware config specifies GPU type, count, nodes
  - Framework choice locked in config
  - Generated train.py reads config, applies no logic
- **Priority**: HIGH
- **Phase**: 2

### R-SCAFFOLD-4: modules/ Directory Structure
- **Title**: Standardize custom code organization in setup
- **Acceptance**:
  - `modules/rewards/` for custom reward functions
  - `modules/adapters/` for custom framework adapters (if needed)
  - `modules/__init__.py` auto-discovers + registers plugins
  - Can be empty if using built-in rewards/frameworks
- **Priority**: MEDIUM
- **Phase**: 2

### R-SCAFFOLD-5: Standard Setup File List
- **Title**: Every setup_{name}/ contains exact set of files
- **Acceptance**:
  - requirements.txt (framework + technique specific)
  - config.yml (copy of input)
  - train.py (executable)
  - prepare.py (executable)
  - run.sh (bash wrapper: prepare → train)
  - modules/ (empty or with custom code)
  - .cache/ (generated by prepare.py)
  - .env.example (optional secrets template)
  - No extraneous files
- **Priority**: HIGH
- **Phase**: 2

## Tradeoffs & Constraints

| Item | Constraint | Rationale |
|------|-----------|-----------|
| Scaffold Minimal | Yes — no bells & whistles | Goal: starter template, not final product |
| Config 100% truth source | Yes — zero env var magic | Reproducibility, traceability |
| Plugins all retained | Yes — even if "dead code" | Dynamic registration, extensibility |
| SkyPilot optional | Yes — local fallback OK | Not all users use cloud |
| Type hints extended? | No — keep existing | Focus on functionality, not refactoring |
| Framework adapter refactor? | No — keep working code | High risk, low priority |
